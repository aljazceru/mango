## Spacing Scale

Values are in points (iOS/Android) or pixels (Desktop). All values are from the standard set {4, 8, 16, 24, 32, 48, 64}.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4 | Icon gaps, inline padding, section label bottom margin |
| sm | 8 | Inter-element spacing inside a card, between badge and text |
| md | 16 | Card internal padding (horizontal and vertical), row vertical padding, horizontal screen margin |
| lg | 24 | Vertical spacing between major sections, bottom safe-area padding |
| xl | 32 | Bottom scroll buffer |
| 2xl | 48 | Large vertical gaps (reserved, not used in this phase) |
| 3xl | 64 | Full-screen vertical gap (reserved, not used in this phase) |

Exceptions: Touch targets for tappable rows must be at least 44pt tall on iOS and 48dp tall on Android (platform accessibility minimum). Desktop button padding `[8, 16]` maps to 44px effective height at 14px font size using `sm` vertical and `md` horizontal.

Source: Observed from existing codebase patterns in SettingsView.swift, SettingsScreen.kt, and views/settings.rs. Non-standard values 12 and 20 removed тАФ not in the required set {4, 8, 16, 24, 32, 48, 64}.

## Color

### Shared Semantic Roles

Colors differ by platform because iOS uses a blue accent and Desktop/Android use teal. Both are established тАФ do not harmonize them. The semantic role is the same; the hex differs.

| Role | iOS dark | iOS light | Desktop/Android dark | Desktop/Android light | Usage |
|------|----------|-----------|---------------------|----------------------|-------|
| Dominant (60%) | `#1A1A1A` | `#F7F7F7` | `#0E0E10` | `#F7F7F7` | Page background |
| Surface (30%) | `#262626` (list bg) | `#EDEDED` | `#181820` | `#FFFFFF` | Card containers, nav chrome |
| Secondary surface | system grouped | system grouped | `#26262A` | `#EEEEEE` | Section backgrounds |
| Accent (10%) | `#4D9EFF` | `#1A6FD4` | `#00C896` | `#008C64` | See reserved list below |
| Destructive | `#E53E3E` | `#B71C1C` | `#E53E3E` | `#B71C1C` | Delete/Remove actions only |
| Success | `#00C896` | `#008C64` | `#00C896` | `#1B5E20` | Enabled badge, healthy status |
| Warning | system yellow | darker yellow | `#F2BF1A` | `#7B5800` | Degraded, expired states |
| Muted text | `#8C8C8C` | `#737373` | `vc.muted` `#787880` | `#8C8C8C` | Descriptions, placeholders, memory count numeral |

Accent reserved for:
- Primary save/enable action buttons (filled state)
- Active section-level navigation links when focused
- "Set Default" badge on active provider
- "Configured" status indicator on Brave API key row (when key is set)

Accent is NOT used for: section headers, destructive buttons, regular row text, dividers, cards, memory count numeral.

Memory count numeral color: muted (`#8C8C8C` / `onSurfaceVariant` / `vc.muted`). This is the canonical color тАФ the interaction states section and the section layout contract both confirm muted. Accent is not used for the memory count.

Destructive color is used ONLY for: Remove Backend button, Delete Memory (accessed from Memories screen, not Settings), confirmation dialogs.

### Focal Point

The **PROVIDERS section** is the primary visual anchor of the Settings screen on all three platforms. It appears first, contains the most visually dense content (provider cards with status badges), and is the most frequently interacted-with section. No special visual treatment is added тАФ its position, density, and existing status badges naturally draw the eye. The two new sections (MEMORY, TOOLS) are secondary anchors positioned mid-screen.

Source: AppColors.swift, theme.rs view_colors(), existing SettingsScreen.kt color usage.

## Section Layout Contract

The Settings screen section order is locked as follows. This is the authoritative order for all three platforms:

```
1. PROVIDERS     тАФ existing, unchanged (primary visual anchor)
2. DEFAULTS      тАФ existing, unchanged
3. MEMORY        тАФ NEW: navigation row to Memories screen + count badge
4. TOOLS         тАФ NEW: Brave Search API key field
5. APPEARANCE    тАФ existing, unchanged
6. [Advanced]    тАФ existing, collapsible, unchanged
```

### MEMORY Section

**Content:** One tappable row тАФ "Memories" тАФ that dispatches `PushScreen(screen: .memories)`.

Row anatomy:
- Left: icon + label "Memories" (SF Symbols `brain` on iOS; text only on Android/Desktop)
- Right: memory count numeral in muted color (hidden when count is 0), then chevron/arrow indicator

Interaction: full-row tap. No swipe actions on this row.

State variants:
- Count = 0: Show row label only, no count badge. Chevron still visible.
- Count > 0: Show count as a secondary label (not a filled badge тАФ just a number in muted text).

**iOS specific:** Use a `Button` (not `NavigationLink`) inside `Section("Memory")` that dispatches `.pushScreen(screen: .memories)`. Follows the existing routing pattern тАФ see RESEARCH.md Pattern 1.

**Android specific:** Use a `Card` with a `Row` inside that has `Modifier.clickable`. Consistent with existing card pattern. Not a `Card(onClick = ...)` overload тАФ use `Modifier.clickable` on the inner Row for safe compatibility with the project's Material3 version. Row padding: `md: 16` (16dp).

**Desktop specific:** Use a `button(...)` wrapping a `row![label, space, count, chevron_text]` styled as `vc.card` container. Follows existing `ghost_btn` / container pattern from views/settings.rs. Button padding: `[8, 16]` (vertical `sm: 8`, horizontal `md: 16`).

### TOOLS Section

**Content:** One Brave Search API key input field with a Save API Key button.

Row anatomy:
- Sub-header: "Web Search" (secondary label, not a section header)
- Description text: "Required for agent web search tool. Keys are stored locally and never sent to third parties."
- Secure text field (masked input, always starts blank on screen load)
- Placeholder text: "Enter Brave Search API Key" when not configured, "Key configured тАФ enter new key to update" when `brave_api_key_set == true`
- Save API Key button: enabled only when input field is non-empty after trim

**iOS specific:** Use `SecureField` + `Button("Save API Key")` inside `Section("Tools")`. Add `@State var braveApiKeyInput: String = ""`. Save dispatches `.setBraveApiKey(apiKey: braveApiKeyInput)` and clears the field.

**Android specific:** Use `OutlinedTextField` with `PasswordVisualTransformation` + `Button("Save API Key")`. Follows existing API key field pattern for provider keys (uses `label` composable for placeholder text, consistent with existing `OutlinedTextField` usage for provider API keys at line 537). Field padding: `md: 16` (16dp). Note: Material3 `OutlinedTextField` `label` floats when focused/filled, which differs from iOS `SecureField` placeholder behavior -- this is intentional platform-specific UX.

**Desktop specific:** Use `text_input(...).secure(true)` + `action_btn("Save API Key", ...)`. Follows existing `key_input` pattern. Save button enabled only when `!brave_api_key_input.trim().is_empty()`.

### Home Screen Change

The "Memories" toolbar button is removed from the home screen on all three platforms.

- **iOS:** Remove the `Button("Memories")` from `ContentView.homeView` toolbar `HStack`.
- **Android:** Remove the `TextButton(onClick = PushScreen(Screen.Memories))` from `MainApp.kt` topBar actions.
- **Desktop:** Remove the equivalent Memories navigation element from `views/home.rs` toolbar (confirm exact location during implementation).

"Documents" and "Agents" toolbar buttons remain. "Settings" button remains. Only "Memories" is removed.

| State | Visual |

### Memory Row

| State | Visual |
|-------|--------|

## Platform-Specific Notes

### iOS
- Uses SwiftUI `List` with `Section` тАФ existing structure. New sections insert between `defaultsSection` and `appearanceSection`.
- Memory row: `Button` inside `Section("Memory")` dispatching `.pushScreen(screen: .memories)`.
- Brave key: `SecureField` inside `Section("Tools")`.
- No `NavigationLink` тАФ app uses custom push-pop router via `appManager.dispatch`.

### Android
- Uses `LazyColumn` with `item {}` blocks тАФ existing structure. New items insert between DEFAULTS item block and APPEARANCE item block.
- Memory row: `Card` containing a `Row` with `Modifier.clickable`. Row padding: 16dp (`md`).
- Brave key: `OutlinedTextField` with `PasswordVisualTransformation` inside a `Card`. Field padding: 16dp (`md`).
- Section header labels: `Text("MEMORY", style = MaterialTheme.typography.labelSmall, color = onSurfaceVariant)` тАФ identical to existing `PROVIDERS` / `DEFAULTS` headers.
- Note: `OutlinedTextField` uses the `label` composable for placeholder text (consistent with existing provider API key fields). The Material3 floating label behavior differs from iOS `SecureField` placeholder -- this is intentional platform-specific UX.

### Desktop (iced)
- Uses `column![]` composition in `views/settings.rs`. New sections inserted between `defaults_content` and `appearance_row` in the compose block.
- Memory row: `container(button(...))` styled as `vc.card` with `border: Border { radius: 8.0.into() ... }`. Button padding: `[8, 16]` (vertical `sm: 8`, horizontal `md: 16`).
- Brave key: `text_input(...).secure(true)` + `action_btn("Save API Key", ...)` inside a `container` styled as `vc.card`.
- `view()` function signature must gain `brave_api_key_input: &'a str` as a new param. `brave_api_key_set` and `memory_count` are read from the existing `state: &'a AppState` argument (no separate params needed -- cleaner API surface).
- New `Message` variants needed: `SettingsBraveApiKeyChanged(String)` and `SettingsSaveBraveApiKey`.
